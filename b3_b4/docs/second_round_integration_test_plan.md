## 1. 目的

在第一次联调「跑通数据交接」的基础上，本轮目标是**验证完整闭环链路的端到端正确性与稳定性**，
并首次纳入**容量预测结果反馈 A 组做异常复盘**的环节（第一次联调时明确顺延到本轮）。

本轮开始引入**量化验收门槛**（字段符合率、链路成功率、端到端时延等），
但仍**不做模型精度达标考核**，模型精度评估留待后续实验阶段。

---

## 2. 与第一次联调的差异（本轮新增）

| 维度 | 第一次联调 | 第二次联调（本轮） |
| :-- | :-- | :-- |
| 链路 | A→B1→B2→B4→B3 单向 | **闭环**：新增 B3 → A 异常复盘反馈 |
| 执行 | 样例文件占位为主 | B1/B2 用真实代码产出，B4 支持 `real` 模式实跑 |
| 异常类型 | `cpu_overload` | `cpu_overload` + 新增 `io_bottleneck` |
| 验收 | 字段/命名可解析即可 | 引入**量化门槛**（见 §7） |
| 健壮性 | 初步检查 | 覆盖缺字段/空文件/时间戳断裂/乱序 |

## 3. 联调范围与链路

```text
   A组 正常负载+异常注入 ──► B1 特征分析 ──► B2 调优建议 ──► B4 自动化验证
        ▲                                                        │
        │                                                        ▼
        └──────────── B3 容量预测 (消费 A1 + B4 验证后数据) ◄──────┘
                       │
                       └──►  ★本轮新增：B3 预测结果反馈 A 组做异常/容量复盘
```

- **主链路**：A1 → B1 → B2 → B4 → B3（沿用第一次）。
- **新增闭环**：B3 → A（把容量预测与资源趋势回传 A 组，辅助异常复盘）。

## 4. 环境与前置条件

| 项 | 要求 |
| :-- | :-- |
| 租户 | 独立测试租户（**不使用共享 `test` 租户**），具备 `ALTER SYSTEM/TENANT` 权限 |
| 负载工具 | BenchmarkSQL（TPC‑C）、tpch‑obs（TPC‑H） |
| 采样粒度 | 60 秒，与跨组接口一致 |
| 数据交换 | GitHub Release 目录，按批次异步产出/读取 |
| 历史数据 | A1 提供最近 7 天正常负载 + 至少 2 次异常注入记录 |
| 时间/时区 | `YYYY-MM-DD HH:MM:SS` / `Asia/Shanghai` |

**前置检查清单（联调启动前逐项打勾）：**

- [ ] A 组提供含异常标签的样例数据（≥1 组 `cpu_overload`、≥1 组 `io_bottleneck`）
- [ ] B1/B2 代码可按 v0.1 接口格式输出，或提供合格样例文件占位
- [ ] 独立测试租户已就绪，回滚 SQL 可执行
- [ ] 各组本地脚本自查通过，可读写 Release 目录结构

## 5. 测试场景设计

### 场景一　闭环连通性验证
逐段验证 A→B1→B2→B4→B3→A 每一步文件按约定格式生成、传递、解析，不判业务正确性。

### 场景二　端到端链路验证
独立测试租户中完整跑通一次「异常注入 → 特征分析 → 调优建议 → 自动化验证 → 容量预测 → 复盘反馈」，
产出一份完整端到端记录。异常类型本轮取 `cpu_overload` 与 `io_bottleneck` 各一轮。

### 场景三　异常输入健壮性
构造缺失字段、空文件、时间戳断裂、乱序四类脏数据，验证下游按缺失值规则降级、不崩溃。

### 场景四（新增）　容量预测反馈复盘
B3 输出 `capacity_forecast` 后回传 A 组，验证 A 组能否用预测的资源趋势辅助定位/复盘异常时段。

## 7. 验收标准

### 7.1 接口符合性（硬门槛，必须 100%）

| 指标 | 门槛 |
| :-- | :-- |
| 各接口文件**字段符合率** | = 100%（字段名/类型/命名规则完全匹配接口文档） |
| 文件命名符合率 | = 100% |
| P0 用例通过率 | = 100% |

### 7.2 链路稳定性

| 指标 | 门槛 |
| :-- | :-- |
| 主链路端到端成功率 | ≥ 95%（重复 ≥5 轮） |
| 健壮性用例（场景三） | 全部**降级不崩溃**，无未捕获异常 |
| 高风险参数误自动执行 | = 0 起 |
| 变更缺失 `rollback_sql` | = 0 条 |

### 7.3 端到端时延（记录为主，设软阈值）

| 环节 | 参考软阈值 |
| :-- | :-- |
| A1→B1→B2 | ≤ 10 min |
| B4 单轮验证（simulation） | ≤ 1 min |
| B4 单轮验证（real，含跑 benchmark） | 记录实测，暂不设硬阈值 |
| B3 预测 | ≤ 2 min |

> 时延本轮以**记录+基线**为主，超阈值不判失败，作为后续优化参考。

### 7.4 本轮判定规则

- **通过**：7.1 全部达标 且 7.2 全部达标。
- **有条件通过**：7.1 达标、7.2 有个别 P1/P2 未达标但有明确整改项。
- **不通过**：任一 7.1 硬门槛未达标。

## 9. 风险与应对

| 编号 | 风险 | 影响 | 应对 |
| :-- | :-- | :-- | :-- |
| RK‑01 | 独立测试租户权限/资源不足 | B4 `real` 模式无法实跑 | 先用 `simulation` 跑通，权限到位后补 real 回归 |
| RK‑02 | B2 缺 `rollback_sql`/`current_value` 字段 | 回滚不精确 | B2 补字段，过渡期 B4 用模板占位 |
| RK‑03 | A1 历史数据不足 7 天 | B3 长期(7d)预测样本不足 | 先用现有样例跑通链路，标注低置信；数据积累后回归 |
| RK‑04 | 异常注入影响其他实验 | 干扰他人 | 严格限定独立租户，变更全部可回滚 |
| RK‑05 | 端到端异步交换延迟 | 联调节奏受阻 | 明确各批次 Release 时间点，异步产出 |
