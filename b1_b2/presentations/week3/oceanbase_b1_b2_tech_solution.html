B1 · 参数影响因子分析

目标：把 Top100「命中关键词的候选表」升级为「可解释的证据链」——不只告诉 B2 调哪个参数， 还要说清楚为什么调、在什么 workload 下调。四条证据线并行采集，融合后再排序输出。

四条证据线（关键词 / 文档 Hint / workload 相关性 / 历史重要性）并行采集 → 加权融合 → 输出带边界的 TopK 参数表
参数特征

来自 top100_performance_params_real.csv

  • category 参数分类
  • scope cluster / tenant
  • edit_level 静态 / 动态生效
  • risk_level 风险等级
  • value_range 取值范围
Workload 特征

三类真实 workload

  • 轻量 SQL（连通性 / 小样本验证）
  • BenchmarkSQL / TPC-C（写入密集）
  • TPC-H 22 查询（大查询 / 内存密集）
  • 对应运行指标向量
性能特征

来自真实压测结果

  • avg / P95 / P99 latency
  • qps / tps 吞吐
  • error_count 错误数
  • 后续补 CPU / 内存 / IO / 等待事件

Top5 参数假设（供后续实验验证）

参数类型机制假设优先关注 workload
cpu_quota_concurrency数值并发调度资源分配TPC-C 并发排队 / TPC-H 大查询执行
memstore_limit_percentage百分比内存水位、冻结行为TPC-C 写入密集路径
clog_io_isolation_mode枚举日志 IO 隔离TPC-C 写事务日志路径
enable_sql_audit布尔观测能力 vs 额外开销权衡两类 workload 均需评估开销
large_query_worker_percentage百分比大查询工作线程资源TPC-H 大查询执行

B1 输出接口（供 B2 直接消费）

# b1_parameter_candidates_exp001.csv param_name, category, scope, edit_level, risk_level, value_range, evidence_score, hint_source, hint_confidence, applicable_workload # b1_feature_matrix_exp001.csv workload_metrics + param_values + performance_targets # b1_feature_importance_exp001.csv param_name, importance_score, method, workload # b2_param_search_space_exp001.yaml —— B1 → B2 的交接文件 topK: [param_name, min, max, type, step, scope, edit_level, risk_level]
局限与应对 当前既无 OceanBase 调优文档语料，也无历史参数实验样本：文档 Hint 先用规则 + LLM 抽取轻量版， 历史重要性列先留空占位，待真实参数实验产出后逐步回填，避免用空数据伪装成成熟模型。

B2 · 深度强化学习智能调优

目标：把调参建模为连续控制问题，agent 只在 B1 缩小后的安全动作空间内闭环搜索—— 不对 OceanBase 全参数空间做 RL，也不在共享租户上直接放开探索噪声。

state → agent（actor / critic, DDPG）→ action → environment（OceanBase 实例）→ reward → 回到 state；每一步 transition 写入 replay buffer，供下一轮训练采样
State · 三层

workload + 性能 + 系统指标

  • workload one-hot（轻量 SQL / TPC-C / TPC-H）
  • 性能指标：avg/P95/P99、qps/tps、error_count（已具备）
  • 系统运行指标：CPU / 内存 / 磁盘 IO / 等待事件 / SQL audit 聚合（待 A 组数据补齐）
Action · 安全空间

初期仅 Top5，经过 sanitizer

  • 范围裁剪至 B1 给出的 min/max
  • 连续 / 枚举 / 布尔分别编码
  • 只允许低 / 中风险；can_modify=UNKNOWN 先屏蔽
  • 每次动作生成 rollback 配置
冷启动策略

样本不足时的过渡方案

  • OtterTune 风格 GP / 贝叶斯优化作为 baseline
  • 逐步引入 DDPG 主线并与之对比
  • 离线可计算 reward 先行，真实执行结果接入后再校准

Reward 函数设计

reward = 0.45 * throughput_gain + 0.35 * latency_reduction_p95 + 0.10 * latency_reduction_p99 - 0.10 * error_penalty - safety_penalty # 基线参照点 TPC-C tps=1.826 p95=716.0 ms TPC-H qps=6.5269 p95=192.592 ms
  • 范围

    连续参数示例：query_memory_limit_percentage [0,100]、log_disk_utilization_threshold [10,100)、io_scheduler_thread_count [1,16] —— 用 DDPG 直接输出连续向量，再做边界裁剪与类型转换。

  • 生效

    参数生效方式不同：动态 / 静态生效，cluster / tenant scope 差异，均需写入 action mask，避免生成不可执行或需重启的动作。

  • 安全

    共享 test 租户当前不可直接改参：短期内以离线设计、仿真 reward 为主，待开放安全执行窗口后再接入真实执行结果做闭环训练。

B2 输出接口 & 复用组件

# 输出 parameter_config_{experiment_id}.csv parameter_performance_{experiment_id}.csv b2_tuning_recommendations # replay experience { state, action, reward, next_state } # 可复用的 controller 接口 src/b2_tuning/env.py src/b4_validation/param_applier.py src/b4_validation/experiment_tracker.py

OceanBase 官方方案对比与学术方案补充价值

B1/B2 不是要替代官方调优流程，而是要补上官方流程里"经验依赖、难复用、非自动化"的缺口。 先看清官方方案能做到什么、做不到什么，再确定学术方法该补在哪一环。

官方调优 / 诊断流程
  • 01

    观察 workload 与业务目标 —— 明确当前是 OLTP 写入密集还是 OLAP 大查询场景，确定优化目标（延迟 / 吞吐 / 稳定性）。

  • 02

    采集运行数据 —— SQL 审计、系统指标（CPU / 内存 / IO）、等待事件、租户资源使用情况。

  • 03

    定位瓶颈 —— 逐项排查 CPU、内存、IO、SQL 执行计划、日志写入、租户资源限制中的具体瓶颈点。

  • 04

    人工调整参数 —— 依据官方文档与最佳实践手册，由 DBA 凭经验决定改哪个参数、改到什么值。

  • 05

    压测验证与固化 —— 执行压测观察效果，效果不佳则回滚，效果达标则固化配置。

官方方案优势

稳、准、可解释

  • 流程稳定、安全，改动可追溯
  • 与 OceanBase 参数体系、运维工具天然一致
  • 适合线上问题定位、人工审核与低风险变更
官方方案局限

依赖经验，难以复用与自动化

  • 高度依赖 DBA 个人经验
  • 参数多，人工筛选成本高
  • 历史调优数据难以沉淀复用
  • 缺少自动化搜索与 workload-specific 参数推荐
  • 对 B1/B2 这种智能调优闭环支持不足
学术方案的补充价值
DB-BERT

文档 → 结构化 Hint

  • 把官方文档 / 调优手册转成可计算的 parameter hint
  • 补上"参数怎么选"的经验起点
OtterTune

历史数据 → 重要性排序

  • 用历史调优数据做 workload characterization
  • 补上"哪些参数真正重要"的统计依据
CDBTune

安全空间 → 连续搜索

  • 深度强化学习在受限动作空间内搜索连续配置
  • 补上"自动闭环搜索"这一环
组合方案 官方规则兜底 + 学术算法推荐 + 人工审核执行 —— B1/B2 产出的建议始终经过官方安全边界过滤，最终变更仍由人工审核确认，而不是替代 DBA 决策。

跨组协同与接口对齐

B1/B2 不是孤立模块：输入依赖 A 组的异常检测数据，输出会影响 B3 的容量预测，执行落地依赖 B4 的实验基础设施。 接口对齐得越早，返工成本越低。

B1/B2 × A 组

输入依赖

  • 需要 normal_load、system_tables、anomaly_labels
  • 需要 CPU / memory / disk / session / wait event / SQL audit 等系统指标
  • 这些指标是 B2 State 三层中"系统运行指标"的唯一数据来源
B1/B2 × B3

双向影响

  • B1 输出 feature_importance，可辅助 B3 容量预测的特征筛选
  • B2 的调优建议会改变资源使用，需反馈给 B3 重新评估容量变化
B1/B2 × B4

执行依赖

  • B4 提供自动化 benchmark、参数应用、实验追踪能力
  • B2 输出的 recommendation 需由 B4 执行、验证、记录 rollback
  • 复用 src/b4_validation/param_applier.py 与 experiment_tracker.py

跨组数据规范

# 主线标识 以 experiment_id 作为跨模块关联主键 # 统一文件命名 parameter_config_{experiment_id}.csv parameter_performance_{experiment_id}.csv # 后续 release 目录需补齐 README_dataset.md data_schema.yaml
06

本周可落地结论与完成度

四条结论按优先级排列，决定了接下来接口开发与实验设计的顺序；完成度小结用于同步本周四项任务的实际进度。

01

B1 要给出证据链，而不是一份列表。输出参数元信息、文档 hint、workload 相关性、历史实验重要性的组合，让 B2 知道「为什么调」。

02

B2 只接收 B1 筛出的安全 TopK。不对 OceanBase 全参数空间做 RL，先围绕第二周 Top5 建模并验证闭环。

03

当前 baseline 已足够定义 reward 参照点。TPC-C 以吞吐、P95/P99 延迟为主；TPC-H 以总耗时、平均延迟与尾延迟为主。

04

官方规则 + 学术算法 + 人工审核三者组合，而非互相替代。B1/B2 的角色是补上官方流程里"难复用、非自动化"的缺口，不是绕开安全边界自行决策。
